# ai-agents-conventions 项目介绍

## 一句话

这是一个**跨项目的 AI 协作约定仓库**：集中存放通用规则（`Agents.md`）和可复用的 Grok Skills，让人和 AI 在多个业务仓库里用同一套工作方式，而不是每次会话重新约定。

本仓库**不包含**业务产品代码、接口实现或页面工程。

---

## 解决什么问题

在多个 monorepo / 多仓库并行开发时，常见问题是：

| 问题 | 后果 |
|------|------|
| 协作习惯只存在于某次对话里 | 换会话、换人后约定丢失 |
| 只有总仓说明，子项目没有自身上下文 | AI 凭「常见项目长什么样」猜架构 |
| UI 出稿方式不统一 | 套娃原型、死图冒充可审查稿、默认全量切图 |
| 多项目一起改却不先盘点文档 | 改错仓、契约对不上、补丁叠补丁 |

本仓库用三层约定把上述问题钉到磁盘上：

1. **通用规则**（`Agents.md`）：怎么思考、怎么改代码、交付给谁看  
2. **通用 Skills**：怎么出 UI、怎么恢复多项目上下文、怎么一次完成 uni-app 三端兼容
3. **登记与模板**：业务仓如何自备 `PROJECT_CONTEXT` 与项目 skill  

业务细节仍写在各自业务仓里；这里只沉淀**可复用的工作方式**。

---

## 适合谁用

- 用 AI 编程助手（如 Grok）做日常开发的个人或小团队  
- 同时维护后端、管理端、H5/小程序等多个子项目的 monorepo  
- 需要稳定 UI 原型 / 切图 / Inspect 交付标准的产品与前端协作  
- 希望「新会话打开就能按清单恢复上下文」的负责人

## 核心内容说明

### 0. skills/ — Skill 总目录

- **AI 先读** [`skills/README.md`](../skills/README.md)，再打开命中的 skill 正文  
- 实现文件在 `.grok/skills/<name>/`  
- 新增流程：[`skills/ADDING.md`](../skills/ADDING.md) + 模板 `skills/_template/`  
- 未在索引表登记的，不视为本仓正式 skill  

### 1. Agents.md — 通用协作规则

会话级总闸门，约定例如：

- 使用中文沟通  
- 第一性原理、根因解决、挑战方案假设  
- 先理解调用链再改；交付说明改了什么、如何验证  
- **最终读者视角**：用户可见成品不写作者/评审/制作过程  
- **公开 UI 文案闸门**：页面正文只保留用户能感知的价值与动作  
- **多项目上下文闸门**（≥3 个独立项目时强制）：先公共登记表，再各项目 docs/skill，缺失先补再改  

完整条目以仓库根目录 `Agents.md` 为准。

### 2. ui-standard — 通用 UI 出稿规范

约束「以后任意 UI 怎么出」，**不绑定**某一业务页或旧稿。

**覆盖端（同等闸门）**：PC / 管理后台 / 桌面 Web、H5、小程序及多端对照。  
不得因「是 PC 后台」就改用死图或跳过 Inspect。

覆盖要点：

- 单文件真实 DOM 原型（禁止多层 iframe 套娃）  
- 多端对照时业务信息一致，仅导航壳与信息密度可不同  
- 可审查 Inspect（点选量尺寸/颜色/边距，禁止整页 PNG 冒充；**PC 表格/侧栏同样适用**）  
- Figma 导出与限流时的本地降级  
- **不规则运行时切图：必须先问用户，同意后再导出**  

使用前阅读：`.grok/skills/ui-standard/SKILL.md`  
交付前自检：`.grok/skills/ui-standard/references/checklist.md`

### 3. project-context — 多项目上下文闸门

当管辖范围内有 **≥3 个独立项目** 时，恢复上下文或跨项目任务前必须：

```
读公共 REGISTRY
  → 对比每个项目的 PROJECT_CONTEXT + skill 是否齐全
  → 缺失则深度理解代码后创建
  → 再读当次相关项目文档
  → 再改业务代码
```

配套物：

| 文件 | 用途 |
|------|------|
| `docs/context/REGISTRY.md` | 公共登记表：记路径与齐全状态 |
| 各项目 `docs/PROJECT_CONTEXT.md` | 该项目定位、栈、约定、风险 |
| 各项目 `SKILL.md`（或等价路径） | 在该项目里怎么安全干活 |
| `TEMPLATE_*` | 新建文档时的骨架 |

使用前阅读：`.grok/skills/project-context/SKILL.md`

### 4. uniapp-cross-platform — uni-app 三端兼容闸门

面向 App（Android/iOS）、微信小程序与 H5 共用功能。编码前锁定工具链、渲染模型和平台能力矩阵；平台差异集中在 adapter；交付时分别记录构建、开发者工具、真实浏览器和真机证据。

内容源：[`donaixiami/uniapp-cross-platform-skill`](https://github.com/donaixiami/uniapp-cross-platform-skill)。本仓通过 Git submodule 固定版本，不直接维护 Skill 正文。

- `SKILL.md`：强制工作流、编码红线和完成口径
- `references/compatibility-matrix.md`：模板、生命周期、API、权限、文件、样式和条件编译差异
- `references/checklist.md`：编码前到发布验收的逐项闸门

使用前阅读：`.grok/skills/uniapp-cross-platform/SKILL.md`

## 如何接入业务项目

### 方式 A：整份拷贝（推荐新业务仓）

1. 将 `Agents.md` 拷到业务仓根目录（可与 `AGENTS.md` 同内容）  
2. 将需要的 skill 目录拷到业务仓 `.grok/skills/`  
3. 若 monorepo 子项目 ≥3：在业务仓建 `docs/context/REGISTRY.md`，并为每个子项目补 `PROJECT_CONTEXT` + skill  

### 方式 B：只引用本仓路径

在业务仓的 `Agents.md` 中写明：通用规则与 skill 以本仓库路径为准；会话打开时先读本仓，再读业务仓文档。

### 方式 C：业务仓已有约定

以业务目标为准做合并：本仓提供**出稿方式与协作闸门**；业务产品规格、接口契约、屏幕清单仍归业务 docs 管理。冲突时先对齐负责人目标，再写清取舍。

## 推荐工作流（摘要）

### 普通开发任务

1. 读业务仓 `Agents.md`（或本仓规则）  
2. 读当次相关的 `PROJECT_CONTEXT` 与项目 skill  
3. 改代码 → 验证 → 交付说明  

### UI / 原型任务

1. 读 `ui-standard`  
2. 读业务 UI/产品规范（若有）  
3. 按流水线出原型 / Inspect  
4. 切图：**先问再导**  
5. checklist 自检  

### 多项目 / 恢复上下文

1. 判定项目数是否 ≥3  
2. 读 `docs/context/REGISTRY.md`  
3. 逐项对比 docs/skill  
4. 缺失先补  
5. 再开工  

---

## 本仓明确不做的事

- 不实现具体业务 API、数据库、管理端或小程序页面  
- 不把某一 App 的屏幕清单或演示数据写进通用 skill  
- 不替代各业务仓自己的产品规格与架构文档  
- 不默认在未询问的情况下为业务仓批量导出静态切图  

---

## 维护原则

| 变更类型 | 写哪里 |
|----------|--------|
| 跨项目都适用的协作习惯 | `Agents.md` |
| UI 出稿方式 / 验收闸门 | `ui-standard` |
| 上下文恢复与登记方式 | `project-context` + 模板 |
| 本仓自己的定位变化 | `docs/PROJECT_CONTEXT.md` |
| 工作区有哪些项目、文档是否齐全 | `docs/context/REGISTRY.md`（或业务仓内同等文件） |

业务架构结论、接口契约、模块坑点 → 写在**对应业务项目**的 `PROJECT_CONTEXT` / skill，不要堆进本仓通用文件。
