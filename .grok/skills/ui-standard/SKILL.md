# 通用 UI 出稿规范（本仓库）

用中文沟通。用户可见成品遵守根目录 `Agents.md` / `AGENTS.md` 读者视角闸门（禁止把制作过程写进成品）。

**定位**：约束**今后任意 UI** 的生成与交付方式。  
**不要**把本 skill 写成某个现有业务页的说明书，也**不要**默认复用某一套现成屏幕清单、演示数据或旧文件名。

---

## 何时启用

- 新建或改任何 UI / 原型 / mockup / 视觉稿  
- H5、小程序、双端对照  
- 切图、元素审查（尺寸/颜色/边距）  
- 导出设计资产、写入 Figma  
- 用户说「按规范画 UI」「出界面」「做原型」  

有业务专用 UI 规范文档时：**业务规范管内容与信息架构，本 skill 管出稿方式与质量闸门**；二者冲突时先对齐用户目标，再写清取舍。

---

## 权威输入（每次开干前）

1. **用户当轮目标与验收标准**（页面范围、端、是否要 inspect / Figma）  
2. **本需求相关的产品/UI 规范文档**（若有则必读；没有则先最小补文档再画）  
3. **设计 Token 与组件约定**（项目内 DESIGN / UI_SPEC；若无则先建最小 token 表）  
4. 修改旧稿时：先读旧稿结构，再动刀  

禁止跳过规范、凭「常见 App 长什么样」直接开画。

---

## 标准交付物（按需组合）

| 交付物 | 何时要 | 要求 |
|--------|--------|------|
| 主交互原型（单 HTML） | 默认 | 真实 DOM，可点可切换关键流 |
| 可审查 Inspect 版 | 要量尺寸/颜色/边距/切组件时 | 点选元素出属性面板，非整页死图 |
| 规范/Token 摘录 | 新视觉或新模块 | 色板、字号、间距、状态语义可查 |
| 导出包（SVG/PNG） | 要切图或给设计工具 | 图标矢量优先；屏可为 @2x 参考 |
| 不规则运行时切图 | **仅当用户明确同意后** | 见下文「不规则资源出图」；禁止默认全量导出 |
| Figma 文件 | 用户明确要求 | 能打开；尽量可点选图层，说明局限 |

路径由当次任务决定，写入用户指定目录；未指定时放在对应子项目的 `docs/design/` 或该端的 `static/` 约定目录；**文件名按功能命名，不要写死旧业务文件名**。

---

## 出稿流水线（固定顺序）

```
1. 目标：范围、端、状态、验收
2. 规范：读产品/UI 文档 + token
3. 结构：单文件真实 DOM 原型（禁止多层 iframe 转义）
4. 双端：需要则 H5 | 小程序并排、业务内容一致
5. 交互：关键态齐全；危险操作用显式确认标识
6. 数据：演示数据自洽；金额/数量格式统一
7. Inspect：需要审查时做点选量测版
8. 【可选·须询问】不规则视觉 → 运行时图片资源
9. 导出/Figma：按需；限流则降级本地 Inspect
10. checklist 自检 + 中文交付说明
```

## 不规则资源出图（运行时切图）— **必须先问用户**

### 硬性门禁

原型里大量效果用 **CSS 叠层**（渐变、异形、玻璃、复合阴影）实现；真项目（尤其小程序）往往**不能也不该**把每一层都用样式硬抄。

但：

1. **不要默认**一做完原型就导出全套切图。  
2. **必须先询问用户**是否需要「把不规则视觉导出为项目可用图片」。  
3. 仅当用户明确说「要」「导出」「生成图片资源」「方便直接用」等同意后，再执行导出。  
4. 用户拒绝或未表态 → **跳过**，原型/Inspect 仍可交付。  
5. 询问时用简短选项，例如：  
   - A）导出不规则装饰/底板/背景图到 `static` / 共享资源目录  
   - B）暂不导出，继续用 CSS  
   - C）只导出某几类（背景 / 图标 / 装饰底板）  

### 什么该做成图（用户同意后）

适合出图、不适合在业务页硬仿的：

- 整页**复合背景**（多段线性 + 径向高光等）  
- **异形品牌标**、字标块  
- 业务/Tab/**语义图标**（禁止页面写系统 emoji 当图标）  
- 难对齐的**装饰底板**（Tab 选中盘、图标浅色盘、FAB 底、横幅渐变底等）  
- 无 blur 时的**玻璃卡/详情头图**替代底图  
- 平台壳装饰（如小程序胶囊占位图）  
- 默认头像渐变盘（无用户上传图时）  

### 什么继续 CSS/组件（默认不要切图）

- 普通白卡片、列表行、输入框、分段器  
- 文案、金额、状态标签结构  
- 间距、圆角体系、安全区、flex 布局  
- 能用纯色/简单渐变稳定表达的按钮与 pill（除非用户点名要图）  

原则：**结构与状态用代码；难复现、跨端易漂的视觉用图。**

### 执行方式（同意后）

1. 对照当次原型/规范，列出拟导出清单（id、用途、建议尺寸），再导出（或按用户已选范围导出）。  
2. **物理资源默认写入仓库根** `ui-assets/`（全仓共享，不是某一子项目私有）。  
   - 分类示例：`ui-assets/backgrounds/`、`ui-assets/icons/`、`ui-assets/decorations/`  
   - 更新 `ui-assets/README.md` 或 index（路径、用途、是否可用 CSS 替代）。  
3. **各端运行时映射（禁止 junction/快捷方式）**：  
   - **唯一 Git 源**：根目录 `ui-assets/`  
   - uni-app 等：dev/build 时**物理复制**到端内 `static/ui-assets`（构建插件或 `scripts/sync-ui-assets.mjs`），代码仍写端约定路径；副本可 gitignore  
   - 其他端：构建复制或按需引用同一根目录，禁止维护分叉副本  
4. 导出/同步脚本放在本仓库约定位置（如 `scripts/export-irregular-assets.mjs`、`scripts/sync-ui-assets.mjs`），输出指向仓库根 `ui-assets/`。  
5. 交付时说明：根目录路径 + 同步方式 + **哪些仍应用 CSS**。  
6. 禁止把整页截图当作唯一运行时资源；禁止把可维护的布局切成一张死图。

## 不可违反的出稿规则

### 1. 结构

- **单 HTML（或清晰模块化静态文件）**，内联或本地资源即可预览。  
- **禁止**多层 `iframe` + HTML 实体层层转义的「套娃原型」。  
- **禁止**必须连后端/CDN 才能看静态结构（业务接口联调除外）。  
- 用户路径上的界面：**禁止**作者备注、验收话术、技术选型、设计过程等元叙事。  
- 壳层工具（目录、审查器）可以有操作说明，但不得泄漏进「最终用户屏幕」正文。

### 2. 视觉系统

- 使用项目**统一 Token**（色、字号、间距、圆角、阴影），禁止每页一套魔法数。  
- 新模块若无 token：先补最小 token 表，再画界面。  
- **状态不能只靠颜色**（成功/等待/危险等必须有文字或图标语义）。  
- 主信息层级清晰：标题、金额/主指标、次要说明、操作。  
- 图标：统一 SVG（或项目图标体系），**禁止**用 emoji/杂字符冒充业务图标与 Tab 图标。

### 3. 尺寸与平台

- 写明设计基准（如 750rpx / 375 宽）。  
- 多端时：**业务信息一致**，仅适配壳差异（导航、胶囊、安全区、FAB 等）。  
- 修改旧稿时：用户要求「保持原大小/风格」则尊重；否则向规范 token 与可读性靠拢，并说明取舍。

### 4. 交互与状态

- 覆盖任务需要的：默认态、加载/空/错（若规范要求）、关键弹层/确认。  
- 会改数据或权限的动作：用**显式动作标识**做二次确认（如 `data-action`），**禁止**用按钮文案 `includes` 猜测。  
- 返回、Tab、搜索、纯展开：**不要**二次确认。  
- 列表过长、明细过多：用折叠 + 弹层/抽屉，避免无限撑高首屏。  
- 弹层/抽屉若属于需求：**必须在原型里可打开演示**，不能只写在说明里。

### 5. 数据与文案

- 演示数据**全链路自洽**（同一对象金额、数量、状态在各屏一致）。  
- 金额、百分比等格式统一（如金额固定两位小数）。  
- 文案面向最终用户，语气产品化。

### 6. 可审查 Inspect（量测闸门）

用户要「能量尺寸/颜色/边距/切元素」时，交付物必须是**真实可点 DOM**（或 Figma 真实图层），并满足：

- 点选显示：宽高、margin、padding、圆角、背景/文字色、字号字重、display/flex 等  
- 可选中高亮、复制 CSS/宽高  
- 审查模式与交互导航可切换  
- **禁止**用整页 PNG 冒充「可审查 UI」  

### 7. 导出与 Figma

- 图标、可复用符号：**SVG 矢量**优先；运行时 PNG 按端能力选择。  
- 整页位图仅作对照，不作为唯一设计源。  
- **不规则运行时切图**：见专章，**必须先问再导**。  
- Figma：能复用已有文件则不新建；Starter 页数/MCP 限流时说明限制，**降级为本地 Inspect + 导出包**。  
- 双端对照在 Figma 中应 **成对排列**（若做双端）。

---

## 修改策略

1. 先理解现有结构与调用方式，再改。  
2. 最小必要改动；不借机重做无关模块。  
3. 修根因（数据不一致、不可维护结构、错误确认机制），不叠补丁套娃。  
4. 交付后附：改了什么、怎么打开、怎么验收。

---

## 审查他人/旧 UI 时的输出格式

1. **P0** 规范冲突、不可用、结构不可维护、数据自相矛盾  
2. **P1** 双端错误、关键态缺失、状态表达不清  
3. **P2** 细节与 polish  
4. 做对的地方  
5. 修复优先级与建议交付物（原型 / Inspect / 导出）

---

## 禁止清单

- 无规范开画、每页私自色板  
- 多层 iframe 转义原型  
- 整页截图冒充可点选设计源  
- 状态只靠颜色、图标靠 emoji 糊弄  
- 确认逻辑靠文案包含关系猜测  
- 用户屏里的开发者/评审元叙事  
- 把本 skill 写成「某一个现有 App 页面的复制粘贴模板」  
- **未经用户同意**就批量导出运行时切图 / 覆盖 `static` 资源  
- 把本可 CSS 维护的布局、列表、表单整页切成不可维护的大图

完成前执行 `references/checklist.md`。  
业务专项清单（若有）在业务文档中另行维护，不写入本 skill 正文。
