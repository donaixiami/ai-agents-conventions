# 通用 UI 出稿规范（本仓库）

用中文沟通。用户可见成品遵守根目录 `Agents.md` / `AGENTS.md` 读者视角闸门（禁止把制作过程写进成品）。

**定位**：约束**今后任意 UI** 的生成与交付方式（**含 PC / 管理后台 / 桌面 Web**，不限移动端）。  
**不要**把本 skill 写成某个现有业务页的说明书，也**不要**默认复用某一套现成屏幕清单、演示数据或旧文件名。

**端覆盖（同等闸门）**：

| 端 | 示例 | 是否适用本 skill |
|----|------|------------------|
| PC / 桌面 Web | 管理后台（如 Vben）、运营台、桌面官网 | **是，不得降级** |
| H5 | 手机浏览器、可嵌 WebView 的 H5 | 是 |
| 小程序 | 微信等小程序 | 是 |
| 多端对照 | PC + H5、H5 + 小程序、三端等 | 是；业务信息一致，仅壳差异 |

---

## 何时启用

- 新建或改任何 UI / 原型 / mockup / 视觉稿  
- **PC 管理端、桌面 Web、H5、小程序**，或以上任意组合的多端对照  
- 切图、元素审查（尺寸/颜色/边距）  
- 导出设计资产、写入 Figma  
- 用户说「按规范画 UI」「出界面」「做原型」「画后台页」  

有业务专用 UI 规范文档时：**业务规范管内容与信息架构，本 skill 管出稿方式与质量闸门**；二者冲突时先对齐用户目标，再写清取舍。

---

## 权威输入（每次开干前）

1. **用户当轮目标与验收标准**（页面范围、**哪些端**、是否要 inspect / Figma）  
2. **本需求相关的产品/UI 规范文档**（若有则必读；没有则先最小补文档再画）  
3. **设计 Token 与组件约定**（项目内 DESIGN / UI_SPEC；若无则先建最小 token 表；**PC 与移动可共享语义 token，尺寸阶可分端**）  
4. 修改旧稿时：先读旧稿结构，再动刀  

禁止跳过规范、凭「常见 App / 常见后台长什么样」直接开画。

---

## 标准交付物（按需组合）

| 交付物 | 何时要 | 要求 |
|--------|--------|------|
| 主交互原型（单 HTML） | 默认 | 真实 DOM，可点可切换关键流；PC 用桌面画幅，移动用手机画幅 |
| 可审查 Inspect 版 | 要量尺寸/颜色/边距/切组件时 | 点选元素出属性面板，非整页死图（**PC 同样要求**） |
| 规范/Token 摘录 | 新视觉或新模块 | 色板、字号、间距、状态语义可查 |
| 导出包（SVG/PNG） | 要切图或给设计工具 | 图标矢量优先；屏可为 @1x/@2x 参考 |
| 不规则运行时切图 | **仅当用户明确同意后** | 见下文「不规则资源出图」；禁止默认全量导出 |
| Figma 文件 | 用户明确要求 | 能打开；尽量可点选图层，说明局限 |

路径由当次任务决定，写入用户指定目录；未指定时，UI 原型放在工作区/仓库根目录 `ui-prototypes/`，拆分素材放在根目录 `ui-assets/`，不得默认写入业务子项目的 `docs/design/`、`static/` 或其他运行时目录；**文件名按功能命名，不要写死旧业务文件名**。

## 出稿流水线（固定顺序）

```
1. 目标：范围、端（PC/H5/小程序…）、状态、验收
2. 规范：读产品/UI 文档 + token
3. 结构：单文件真实 DOM 原型（禁止多层 iframe 转义）
4. 多端：需要则按端并排或分页演示；业务内容一致，仅壳/布局密度差异
5. 交互：关键态齐全；危险操作用显式确认标识（PC 表格行操作同样适用）
6. 数据：演示数据自洽；金额/数量格式统一
7. Inspect：需要审查时做点选量测版（含 PC 表格/侧栏/表单）
8. 【可选·须询问】不规则视觉 → 运行时图片资源
9. 导出/Figma：按需；限流则降级本地 Inspect
10. checklist 自检 + 中文交付说明
```

## 原型 UI 与素材根目录归档

本节针对**原型完成后的设计产物整理**，不是把预览 HTML 直接当作业务代码实现。

1. 原型视觉确认后，必须把可复用的图标、背景、装饰、品牌元素、状态素材等从原型中拆分出来，按功能放入根目录 `ui-assets/<feature>/`；原型 HTML 与相关说明放入根目录 `ui-prototypes/<feature>/`。
2. 每个素材至少登记用途、尺寸、格式、适用端、是否可由 CSS/组件替代；禁止只提供整页截图作为前端素材。
3. 根目录是唯一源文件。预览可以相对引用根目录素材，但未经明确同意不得写入任何业务子项目的 `static/`、`public/`、`src/assets/` 或路由入口。
4. 需要运行时使用时，先向用户说明文件清单、目标路径、用途和适用端，获得明确同意后只复制同意的素材；沉默或“先放进去看看”不算同意。
5. 本节不取消“不规则资源出图”专章的先询问门禁；未同意时可以整理根目录设计源，但不得生成或复制项目运行时切图。

## 多端与 PC 专章（同等闸门，壳可不同）

### 硬性一致（所有端）

- 真实可点 DOM / 可审查结构  
- 统一 token 语义（色、状态、层级），禁止每页私自色板  
- 状态不能只靠颜色；图标禁止 emoji 充业务图标  
- 危险/写操作：显式动作标识二次确认  
- 演示数据全链路自洽；用户可见面无作者元叙事  
- 不规则切图：**先问再导**  
- **禁止**因「这是 PC 后台」就改用整页截图、跳过 Inspect、或降低结构可维护性  

### 允许因端而异（壳与布局）

| 维度 | PC / 管理端常见 | H5 / 小程序常见 |
|------|-----------------|-----------------|
| 画幅基准 | 如 1440 / 1280 内容区，或项目约定断点 | 如 375 逻辑宽 / 750rpx |
| 导航壳 | 侧栏、顶栏、多级菜单、Tabs | 自定义导航、TabBar、小程序胶囊 |
| 信息密度 | 表格、筛选条、批量操作、抽屉/对话框 | 列表卡、底栏、半屏 sheet |
| 安全区 | 较少；注意滚动与固定表头 | 底部安全区、刘海、胶囊避让 |
| 主操作 | 工具栏按钮、行内操作、权限码按钮 | FAB、底栏主按钮、页内主 CTA |

多端对照时：

- **同一业务对象**的名称、金额、状态、权限含义必须一致  
- 仅导航壳、密度、控件形态可不同  
- 用户要求多端时：在原型中**成组展示**（并排或可切换），不能只做移动端假装覆盖 PC  
- Figma 多端对照应 **成对/成组排列**（含 PC 画板）

### PC 出稿补充

- 写明设计基准宽度与主要断点（若响应式）。  
- 列表页优先清晰的表头/筛选/空态/加载/权限隐藏，而不是移动端卡片硬拉到桌面。  
- 对话框、抽屉、二次确认在 PC 上必须可点开演示（与移动端 sheet 同级要求）。  
- 管理端权限：演示中可标注「无权限隐藏/禁用」的状态，但**不要**把内部权限码说明书写进用户主工作区正文（可放壳层审查工具）。

## 不规则资源出图（运行时切图）— **必须先问用户**

### 硬性门禁

原型里大量效果用 **CSS 叠层**（渐变、异形、玻璃、复合阴影）实现；真项目（尤其小程序；PC 也有跨浏览器差异）往往**不能也不该**把每一层都用样式硬抄。

但：

1. **不要默认**一做完原型就导出全套切图。  
2. **必须先询问用户**是否需要「把不规则视觉导出为项目可用图片」。  
3. 仅当用户明确说「要」「导出」「生成图片资源」「方便直接用」等同意后，再执行导出。  
4. 用户拒绝或未表态 → **跳过**，原型/Inspect 仍可交付。  
5. 询问时用简短选项，例如：  
   - A）导出不规则装饰/底板/背景图到 `static` / 共享资源目录  
   - B）暂不导出，继续用 CSS  
   - C）只导出某几类（背景 / 图标 / 装饰底板）  
   - D）仅 PC 或仅移动端需要的资源  

### 什么该做成图（用户同意后）

适合出图、不适合在业务页硬仿的：

- 整页**复合背景**（多段线性 + 径向高光等）  
- **异形品牌标**、字标块  
- 业务/Tab/菜单/**语义图标**（禁止页面写系统 emoji 当图标）  
- 难对齐的**装饰底板**（Tab 选中盘、图标浅色盘、FAB 底、横幅渐变底等）  
- 无 blur 时的**玻璃卡/详情头图**替代底图  
- 平台壳装饰（如小程序胶囊占位图）  
- 默认头像渐变盘（无用户上传图时）  
- PC 上难稳定复现的营销头图、复杂插画分隔（普通表格/表单**不要**切图）  

### 什么继续 CSS/组件（默认不要切图）

- 普通白卡片、列表行、**PC 表格与筛选条**、输入框、分段器  
- 文案、金额、状态标签结构  
- 间距、圆角体系、安全区、flex/grid 布局  
- 侧栏/顶栏等布局骨架（结构用代码）  
- 能用纯色/简单渐变稳定表达的按钮与 pill（除非用户点名要图）  

原则：**结构与状态用代码；难复现、跨端易漂的视觉用图。** PC 后台尤以表格/表单结构代码化为先。

### 执行方式（同意后）

1. 对照当次原型/规范，列出拟导出清单（id、用途、建议尺寸、**适用端**），再导出（或按用户已选范围导出）。  
2. **物理资源默认写入仓库根** `ui-assets/`（全仓共享，不是某一子项目私有）。  
   - 分类示例：`ui-assets/backgrounds/`、`ui-assets/icons/`、`ui-assets/decorations/`  
   - 更新 `ui-assets/README.md` 或 index（路径、用途、是否可用 CSS 替代、适用端）。  
3. **各端运行时映射（禁止 junction/快捷方式）**：  
   - **唯一 Git 源**：根目录 `ui-assets/`  
   - uni-app 等：dev/build 时**物理复制**到端内 `static/ui-assets`（构建插件或 `scripts/sync-ui-assets.mjs`），代码仍写端约定路径；副本可 gitignore  
   - PC 管理端：构建复制到 `public/` 或约定静态目录，或打包引用同一根源，**禁止**维护与根目录分叉的第二套资产  
   - 其他端：同上  
4. 导出/同步脚本放在本仓库约定位置（如 `scripts/export-irregular-assets.mjs`、`scripts/sync-ui-assets.mjs`），输出指向仓库根 `ui-assets/`。  
5. 交付时说明：根目录路径 + 同步方式 + **哪些仍应用 CSS** + **覆盖哪些端**。  
6. 禁止把整页截图当作唯一运行时资源；禁止把可维护的布局切成一张死图。

## 不可违反的出稿规则

### 1. 结构

- **单 HTML（或清晰模块化静态文件）**，内联或本地资源即可预览。  
- **禁止**多层 `iframe` + HTML 实体层层转义的「套娃原型」。  
- **禁止**必须连后端/CDN 才能看静态结构（业务接口联调除外）。  
- 用户路径上的界面：**禁止**作者备注、验收话术、技术选型、设计过程等元叙事。  
- 壳层工具（目录、审查器）可以有操作说明，但不得泄漏进「最终用户屏幕」正文。  
- **PC 与移动端同一条结构标准**，不得对后台放宽。

### 2. 视觉系统

- 使用项目**统一 Token**（色、字号、间距、圆角、阴影），禁止每页一套魔法数。  
- 新模块若无 token：先补最小 token 表，再画界面。  
- **状态不能只靠颜色**（成功/等待/危险等必须有文字或图标语义）。  
- 主信息层级清晰：标题、金额/主指标、次要说明、操作。  
- 图标：统一 SVG（或项目图标体系），**禁止**用 emoji/杂字符冒充业务图标、Tab 图标或菜单图标。

### 3. 尺寸与平台

- **必须写明设计基准**（例：PC 1440 内容区；移动 375 / 750rpx）。  
- 多端时：**业务信息一致**，仅适配壳差异（侧栏/顶栏 vs 导航/胶囊/安全区/FAB 等）。  
- 修改旧稿时：用户要求「保持原大小/风格」则尊重；否则向规范 token 与可读性靠拢，并说明取舍。  
- 需求含 PC 时：**不得只交移动端稿**；需求仅移动则可不画 PC，但须在交付说明写清范围。

### 4. 交互与状态

- 覆盖任务需要的：默认态、加载/空/错（若规范要求）、关键弹层/确认。  
- 会改数据或权限的动作：用**显式动作标识**做二次确认（如 `data-action`），**禁止**用按钮文案 `includes` 猜测。  
- 返回、Tab、搜索、纯展开、表格排序筛选展开：**不要**二次确认。  
- 列表过长、明细过多：移动端用折叠 + sheet；PC 用分页/虚拟滚动/抽屉，避免首屏失控。  
- 弹层/抽屉/对话框若属于需求：**必须在原型里可打开演示**，不能只写在说明里。

### 5. 数据与文案

- 演示数据**全链路自洽**（同一对象金额、数量、状态在各屏、**各端**一致）。  
- 金额、百分比等格式统一（如金额固定两位小数）。  
- 文案面向最终用户（C 端用户或 B 端操作员），语气产品化；管理端避免把内部实现术语堆进主界面。

### 6. 可审查 Inspect（量测闸门）

用户要「能量尺寸/颜色/边距/切元素」时，交付物必须是**真实可点 DOM**（或 Figma 真实图层），并满足：

- 点选显示：宽高、margin、padding、圆角、背景/文字色、字号字重、display/flex 等  
- 可选中高亮、复制 CSS/宽高  
- 审查模式与交互导航可切换  
- **禁止**用整页 PNG 冒充「可审查 UI」  
- **PC 表格单元格、侧栏菜单、工具栏按钮**均须可点选审查（若用户要求 Inspect）

### 7. 导出与 Figma

- 图标、可复用符号：**SVG 矢量**优先；运行时 PNG 按端能力选择。  
- 整页位图仅作对照，不作为唯一设计源。  
- **不规则运行时切图**：见专章，**必须先问再导**。  
- Figma：能复用已有文件则不新建；Starter 页数/MCP 限流时说明限制，**降级为本地 Inspect + 导出包**。  
- 多端对照在 Figma 中应 **成组排列**（含 PC 画板，若做了 PC）。

## 修改策略

1. 先理解现有结构与调用方式，再改。  
2. 最小必要改动；不借机重做无关模块。  
3. 修根因（数据不一致、不可维护结构、错误确认机制），不叠补丁套娃。  
4. 交付后附：改了什么、覆盖哪些端、怎么打开、怎么验收。

---

## 审查他人/旧 UI 时的输出格式

1. **P0** 规范冲突、不可用、结构不可维护、数据自相矛盾  
2. **P1** 多端错误（含漏做 PC）、关键态缺失、状态表达不清  
3. **P2** 细节与 polish  
4. 做对的地方  
5. 修复优先级与建议交付物（原型 / Inspect / 导出）

---

## 禁止清单

- 无规范开画、每页私自色板  
- 多层 iframe 转义原型  
- 整页截图冒充可点选设计源  
- 状态只靠颜色、图标靠 emoji 糊弄  
- 确认逻辑靠文案包含关系猜测  
- 用户屏里的开发者/评审元叙事  
- 把本 skill 写成「某一个现有 App 页面的复制粘贴模板」  
- **只约束移动端、对 PC/管理端放水**（需求含 PC 时）  
- 需求明确多端时只交付其中一端且不说明范围  
- **未经用户同意**就批量导出运行时切图 / 覆盖 `static` 资源  
- 把本可 CSS 维护的布局、列表、**表格**、表单整页切成不可维护的大图

完成前执行 `references/checklist.md`。
