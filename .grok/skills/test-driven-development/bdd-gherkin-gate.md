# BDD / Gherkin 闸门（本仓补充）

在遵循上游 [SKILL.md](./SKILL.md)（TDD：先失败测试再实现）的前提下，**写任何正式测试代码之前**，还必须走本闸门。

来源：原截图协作备忘（已改为本仓通用表述；引擎/目录以各业务项目约定为准）。

---

## 硬性顺序

```
1. 禁止先写业务/逻辑实现代码（与 TDD Iron Law 一致）
2. 先做 BDD：只用 Gherkin 风格注释 + 空测试函数名，搭行为骨架
3. 用 AskUserQuestion（或等价「请用户确认」）确认是否继续
4. 用户确认后，再写正式测试体（断言/步骤）
5. 看测试按预期失败（RED）
6. 再写最小实现（GREEN）→ 重构
```

**禁止**：行为注释未写完就写测试体；未确认用户就进入实现；未定义测试就写逻辑代码。

---

## 规则条文（执行用）

### 1. 启用 TDD skill

- 功能、修 bug、重构、改行为：必须使用 `test-driven-development`。  
- **在定义测试之前，禁止编写逻辑/生产代码。**

### 2. 测试框架与目录（项目约定）

- **以当前业务项目的 `PROJECT_CONTEXT` / 项目 skill 为准**（测试框架、命令、目录）。  
- 常见示例（按项目实际选用，不写死）：Jest / Vitest / 框架自带测试，路径如 `*.spec.ts`、`*.test.ts`、`__tests__/` 等。  
- **禁止**把其它无关技术栈（如游戏引擎）的测试目录或工具当作本规范默认。

### 3. 正式测试代码前的 BDD 骨架（强制）

在编写**正式测试代码**（断言、mock 装配、步骤实现）之前，必须先：

1. 遵循 **Behaviour-Driven Development** 实践；  
2. 使用 **Gherkin** 模式，写一个**空测试文件骨架**，其中**仅包含**：  
   - `Given` / `When` / `Then`（及必要时 `And` / `But`）形式的**行为注释**；  
   - **测试函数名**（描述行为，函数体先空或仅占位）；  
3. 使用 **`AskUserQuestion`**（或当前环境等价的用户确认交互）向用户确认是否继续；  
4. **禁止**在完成上述行为注释之前编写任何正式测试代码。

### 4. Gherkin 注释写法参考

- 优先参考**当前项目**内已有示例（`docs` 中的测试约定、既有 `*.spec.ts` / `*.test.ts` 中的 Gherkin 注释等）。  
- 若项目尚无样例：使用最小骨架如下（语法随项目测试框架调整）：

```text
// Feature: <能力一句话>
//
// Scenario: <场景一句话>
//   Given <前置条件>
//   When <用户或系统动作>
//   Then <可观察结果>

describe('<能力>', () => {
  it('<场景：行为描述>', () => {
    // Given ...
    // When ...
    // Then ...
    // （用户确认前：此处不写断言与实现）
  });
});
```

## 与上游 TDD 的关系

| 阶段 | 要求 |
|------|------|
| 骨架 | 本文件：Gherkin 注释 + 空函数名 + 用户确认 |
| RED | 上游 SKILL：写最小失败测试并**看它失败** |
| GREEN | 上游 SKILL：最小实现通过测试 |
| REFACTOR | 上游 SKILL：保持绿的同时整理 |

反模式（mock 测 mock 等）仍读 [testing-anti-patterns.md](./testing-anti-patterns.md)。
