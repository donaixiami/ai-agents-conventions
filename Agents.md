1.

   **语言**：必须用中文回复我。

2.

   **Subagent 使用授权**：当任务适合并行探索、独立验证、拆分实现、专项审查、长流程加速或多视角分析时，AI 可以自主判断并直接调用当前运行环境提供的 subagent/multi-agent 工具，不需要等待用户显式点名或再次确认。使用前先拆清任务边界、输入输出、写入范围、验收标准和主线关键路径；优先把可并行、可独立验证、写入范围清晰且不会阻塞主线的工作交给 subagent。主 agent 必须负责结果整合、冲突处理、事实核验、测试验证和最终交付，不能把未核验的 subagent 结论直接当作最终结论。

3.

   **第一性原理：**不要先套常见方案，先拆解真正目标、已知输入、约束、不可违反条件、事实与假设、最小可行解，以及更稳健的下一层改进，最后给出可执行方案。

4.

   **根因解决**：不要只修表面症状，分析直接问题、根本原因、现有设计为何允许问题发生、哪些临时补丁不推荐，并给出根因级解决方案。

5.

   **挑战用户思路：**不要默认用户判断正确，把用户方案当假设审查，列出隐含假设、最容易错的假设、最可能失败场景、替代方案，并说明负责人视角会怎么选。

6.

   **先理解代码再改：**修改前先找相关文件和调用链，解释当前逻辑如何工作，判断设计为什么这样写，找最小必要改动点，说明风险和验证测试，再开始实现。

7.

   **信任 AI 自主探索：**给目标、验收标准、约束；允许 AI 自主阅读代码、搜索上下文、运行测试、调整实现方案；如果用户给的路径和目标冲突，以目标和验收标准为准；完成后说明改了什么、为什么、如何验证。

8.

   **最终读者视角交付闸门：**创建或编辑任何用户可见的页面、UI、PPT、课程文档、文章或公开内容时，成品必须只面向最终读者/用户，不能把作者/开发者/制作者自己的内部过程写进成品。默认禁止把录制安排、讲师提示、作者备注、前端实现流程、技术选型、设计过程、PPT 制作思路、内部规划等元叙事写入正文；除非用户明确要求。交付前主动检查并清理这些作者视角内容，确保读者看到的是内容、体验、步骤、观点和结论，而不是工作台。

9.

   **公开 UI／视觉稿文案闸门：**创建或编辑用户会看到的页面、HTML mockup、UI 视觉稿、营销页、产品页时，正文不能出现面向设计评审、开发者或内部制作过程的解释性文案。内部推导、设计规范、组件规则、开发边界和验收标准只能放在规范文档或备注中；公开页面只保留用户能直接感知的价值、动作、步骤和结果。

10.

    **记忆**：需要历史背景、长期偏好或旧决策时，可以通过内置记忆或者 Nowledge Mem 先检索再下结论。
    涉及多项目/ monorepo 时，记忆与恢复必须以「公共登记表 + 各项目自身上下文」为准，见第 12 条与 `.grok/skills/project-context/SKILL.md`。

11.

    **通用 UI 出稿规范**：创建或修改本仓库内任何 UI 图、HTML 原型、视觉稿、多端对照（**含 PC/管理端/桌面 Web、H5、小程序**）、切图、元素审查、Figma 导出/写入时，必须先读取并遵循
    `.grok/skills/ui-standard/SKILL.md`（及 `references/checklist.md`）。
    该 skill 约束的是**出稿方式与质量闸门**，不绑定某一现有业务页面或旧稿；具体业务信息架构以当次需求与对应产品/UI 文档为准。
    **PC 端与移动端适用同一套闸门**，不得因「是后台」而降级为整页截图或跳过 Inspect。
    其中「把不规则视觉导出为项目静态图片」属于可选步骤，**必须先询问用户，得到明确同意后再执行**，不得默认全量切图。

12.

    **多项目上下文闸门（≥3 个项目时强制）**：
    当**当前工作目录（或其明确管辖范围）下存在 3 个及以上独立项目**（典型：monorepo 子包、同级多仓库工作区）时，在恢复上下文、开始跨项目任务、或用户要求「读取/记忆上下文」时，必须按以下顺序执行：
    1. 读取**公共上下文登记表**（优先 `docs/context/REGISTRY.md`，或工作区约定的同等文件）；确认是否已登记每个项目的自身上下文路径与 skill 路径。
    2. **逐项目对比**必备物是否齐全：
       - 自身上下文文档：默认 `docs/PROJECT_CONTEXT.md`（或登记表中写明的等价路径）
       - 自身 skill：默认根目录 `SKILL.md`，或 `.grok/skills/<name>/SKILL.md` / 项目内约定 skill 目录
    3. **缺失则先补再干活**：对缺文档或 skill 的项目，先深度阅读代码/README/配置与现有 docs，再创建最小可用的 `PROJECT_CONTEXT` 与项目 skill，并回写公共登记表状态；禁止在缺上下文时靠猜测跨项目改代码。
    4. 已存在则先读公共登记表 → 再读当次相关项目的 `PROJECT_CONTEXT` + skill，再动手。
    详细流程、判定标准、模板与验收见 `.grok/skills/project-context/SKILL.md`。

13.

    **Skills 目录（先索引再正文）**：本仓库可复用 skill 的**总目录与登记表**为 `skills/README.md`。
    AI 在判断任务是否命中某 skill 时，必须先读 `skills/README.md`，再打开表中对应的 `.grok/skills/<name>/SKILL.md`。
    新增 skill：实现放在 `.grok/skills/<name>/`，按 `skills/ADDING.md` 操作，并在 `skills/README.md` 登记；未登记不得当作本仓正式 skill。
    模板见 `skills/_template/`。

14.

    **测试（TDD + BDD 骨架）**：实现功能、修 bug、重构或改行为时，必须使用 `.grok/skills/test-driven-development/`。
    - 在定义测试之前，**禁止**编写逻辑/生产代码。
    - 测试框架与目录以**当前业务项目**的 `PROJECT_CONTEXT` / 项目 skill 为准（勿把某一引擎路径当成全局强制）。
    - 编写**正式测试代码**之前：须先按 BDD，用 Gherkin 的 Given-When-Then 写行为注释 + 测试函数名（空骨架），再用 AskUserQuestion（或等价确认）请用户确认是否继续；**完成行为注释前禁止写测试体**。
    - 细则见 `.grok/skills/test-driven-development/bdd-gherkin-gate.md` 与上游 `SKILL.md`。

15.

    **Git 提交（中文约定式）**：用户要求提交、commit 或推送 GitHub 时，必须遵循
    `.grok/skills/conventional-commits/SKILL.md`。
    格式：`<type>(<可选 scope>): <中文摘要>`；说明默认中文；只暂存相关文件；未同意不 push、不 force。

16.

    **uni-app 三端兼容闸门**：凡使用 uni-app 新建、迁移、修改或审查需要同时运行于 App（Android/iOS）、微信小程序与 H5 的页面、组件、API、权限、文件、样式或第三方依赖，必须先读取并遵循
    `.grok/skills/uniapp-cross-platform/SKILL.md`、`references/compatibility-matrix.md` 和 `references/checklist.md`。该目录是外部仓库 `donaixiami/uniapp-cross-platform-skill` 的 submodule，不在本仓直接编辑 Skill 正文。
    编码前必须锁定目标版本、渲染模型和能力矩阵；交付前必须分别报告三端构建与真机/浏览器验证状态。缺少设备、证书或账号时只能标记“未验证/阻塞”，不得以构建通过宣称三端兼容完成。

17.

    **uni-app 规则同步快捷指令**：用户说“同步 uniapp 规则”时，将业务仓新增且适用于跨项目 App/微信小程序/H5 的规则提炼到独立仓库 `donaixiami/uniapp-cross-platform-skill`，发布后更新本仓 submodule 指针并同步业务仓；仅业务仓专用的规则不提升到通用 Skill。

18.

    **测试服务复用与清理闸门**：测试、联调或验收前需要启动后端、前端、H5、小程序开发服务、Mock、数据库或其他辅助服务时，必须先检查目标端口、健康地址、页面响应、产物状态和进程命令行，确认是否已有当前项目的可用实例。
    1. 已确认当前项目实例健康可用时直接复用，禁止重复启动。
    2. 端口被占用不等于目标服务已启动；若无法确认归属或健康状态，不得误杀，应改用其他端口或报告冲突。
    3. 只有确认目标服务不存在或不可用后才能启动新实例；启动时记录服务名、命令、端口、PID/进程树和启动时间。
    4. 测试正常结束、失败或被中断后，只停止本次任务自己启动且已记录的进程树；用户、其他会话或复用的实例不得停止。
    5. 清理后重新检查进程和端口，确认自己启动的服务已退出；无法安全判定归属时保留进程并向用户说明，不强制杀进程。

19.

    **页面样式还原与原子化迁移闸门**：前端项目开始实现页面时，先使用原生 CSS、SCSS 或 Less 完成页面结构、视觉样式和交互状态，优先保证与设计稿/UI 参考的还原度；不得一开始就为了追求原子化而牺牲还原度。
    1. 页面全部绘制完成并由用户或负责人确认视觉结果后，项目已启用 UnoCSS 或 Tailwind CSS 时，才允许将稳定、已确认的样式迁移为原子化类。
    2. 迁移过程中保持视觉、响应式、交互状态和跨端表现不变；迁移后必须重新检查页面，不得把“已转原子化”当作验收结论。
    3. 组合样式只有在相同语义、相同属性组合在项目中实际使用达到 4 次及以上时，才放入 UnoCSS/Tailwind CSS 的 `shortcuts`、组件类或主题配置；少于 4 次直接在页面使用原子化类，不新增全局配置。
    4. 不能为了凑够 4 次复用而强行抽象，也不能把页面专属样式、一次性布局或仍在频繁调整的样式提前固化到全局配置。
    5. 项目未使用 UnoCSS/Tailwind CSS 时保留经过确认的 CSS、SCSS 或 Less；不得为了遵守本条虚构原子化配置或引入新样式框架。
